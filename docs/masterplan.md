# 프로젝트 센티넬 마스터플랜

> 최초 작성: 2026-02-26

---

## 1. 전체 비전

DS투자증권 리스크관리실의 **리스크포털 프로토타입** 구축 (프로젝트 센티넬).
리스크관리실 내부 업무 전반을 하나의 포털에서 처리할 수 있도록 기능을 순차적으로 확장한다.

---

## 2. 주요 기능 로드맵

| # | 기능 | 상태 | 비고 |
|---|------|------|------|
| 가 | **규정검색** | ✅ 완료 | 조문 검색, PDF 업로드, 법제처 크롤링 |
| 나 | **국내 증권사 재무건전성비율 모니터링** | ✅ 완료 | NCR 순위표, 추이분석, 금감원 FISIS 수집 |
| 라 | **위험관리실무위원회 운영** | 🔲 미구현 | 일정·진행상황·결과 체크 |
| 마 | **캘린더** | 🔲 미구현 | 구글 캘린더 임베딩 |
| 사 | **손익 보고 화면** | 🔲 미구현 | 프런트 부서 손익 입력 → 대시보드·일별 추이 |
| 바 | **스트레스테스트 시뮬레이터** | 🔲 미구현 | 주식·파생·채권 포지션별 시장·유동성리스크 시나리오 손실 점검 |
| 다 | **회사 리스크 대시보드** | 🔲 최후순위 | 포지션 및 리스크 현황 (가장 복잡, 마지막에 구현) |

---

## 3. 보안

**보안이 최우선**이다. 내부 리스크 데이터 및 포지션 정보를 다루므로 기능보다 보안이 먼저다.

- 인증 없이 접근 가능한 화면은 없어야 함
- 민감 데이터(포지션, 손익)는 전송·저장 시 암호화 고려
- 외부 API 키는 환경변수로 관리 (코드에 하드코딩 지양)

---

## 4. 접근 권한

접속자별로 접근 권한을 차등 배정할 수 있어야 한다.

- 예: 리스크관리실 전체 접근 / 프런트 부서는 손익 입력 화면만 / 읽기 전용 계정 등
- 권한 체계는 기능 확장에 따라 세분화

---

## 5. 확장성

- 신규 기능 추가 시 기존 코드·DB에 최소한의 영향만 미쳐야 함
- `views/` 폴더에 뷰 추가 + `app.py` 라우팅 등록만으로 기능 추가 가능한 구조 유지
- DB는 기능별 테이블 분리 (규정검색 / FSS / 향후 손익·포지션 등)

---

## 6. 레이아웃 원칙

어떤 화면이든 동일한 4-패널 구조를 유지한다.

```
┌─────────────────────────────────────────────┐
│                  상단바 (Topbar)              │  ← 로고, 글로벌 메뉴
├────┬────────────────────────┬───────────────┤
│    │                        │               │
│사  │     메인 패널           │  보조 패널    │
│이  │     (Main Panel)        │  (Side Panel) │
│드  │                        │               │
│바  │                        │               │
│    │                        │               │
└────┴────────────────────────┴───────────────┘
```

- **상단바**: 로고, 글로벌 메뉴 (규정검색 / 재무건전성 / 위원회 등)
- **사이드바**: 현재 섹션의 서브메뉴 (아이콘 전용, 70px 고정)
- **메인패널**: 주요 콘텐츠
- **보조패널**: 선택 항목 상세, 원문, 차트 등 컨텍스트 정보
